## Integration Checker Findings

### Blockers (5)

#### BLOCKER-1 — `train.py` → `InstanceDiscriminationModule` crashes
**Affects:** ERA1-01, INFRA-02

`InstanceDiscriminationModule.training_step` expects a 3-tuple `(views, labels, indices)` from `ssl_collate_with_index` + `IndexedDataset`. `train.py` builds `SSLDataModule` with standard `ssl_collate_fn` which yields a 2-tuple `(views, labels)`. Unpacking at line 136 of `methods/instance_discrimination/module.py` raises `ValueError: not enough values to unpack` on the first training step.

**Fix:** `train.py` must detect `method: instance_discrimination` and wrap the dataset in `IndexedDataset` + pass `ssl_collate_with_index` to the DataLoader (as the unit tests do).

---

#### BLOCKER-2 — `eval/linear_probe.py` crashes on launch
**Affects:** EVAL-02

`eval/linear_probe.py` line 221: `dm = SSLDataModule(cfg)` passes a `TrainConfig` Pydantic object as the `data_dir: str` argument. `dm.setup()` calls `os.path.join(self.data_dir, "train")` where `self.data_dir` is a TrainConfig object → `TypeError`.

**Fix:** Change to `SSLDataModule(data_dir=cfg.data_dir, n_views=cfg.n_views, batch_size=cfg.batch_size, num_workers=cfg.num_workers)`.

**Secondary bug (BLOCKER-3 for finetune):** Lines 75-77 use `imgs[:, 0]` (batch-first slice) when SSL batches are `[n_views, B, C, H, W]` (views-first). Correct index is `imgs[0]`. Masked by the TypeError but must also be fixed.

---

#### BLOCKER-3 — `eval/finetune.py` crashes on launch
**Affects:** EVAL-05

Same `SSLDataModule(cfg)` TypeError at line 194. Same wrong view index at lines 87 and 97.

**Fix:** Same as BLOCKER-2.

---

#### BLOCKER-4 — SwAV and DINO trained with wrong crop sizes
**Affects:** ERA2-05, ERA4-02, INFRA-04

`train.py` passes `n_views=8` to `SSLDataModule` standard path, producing 8 crops all at 224×224. `MultiCropDataset` (2×224 + 6×96) exists and is implemented correctly in `core/data.py` but is never instantiated by `train.py`. The `SSLDataModule.train_dataloader()` multi-crop detection branch (`isinstance(self.train_dataset, MultiCropDataset)`) is unreachable via `train.py`.

SwAVModule and DINOModule do not crash — they iterate 8 same-size crops silently — but the global/local crop semantics required by both methods are violated.

**Fix:** `train.py` must detect `method: swav` or `method: dino` and instantiate `MultiCropDataset` with the config's crop parameters before building `SSLDataModule`.

---

#### BLOCKER-5 — SupCon stage-2 trains a randomly-initialized backbone
**Affects:** SUP-01

`train.py` instantiates `SupConFinetuneModule(cfg)` (random backbone) then calls `trainer.fit(model, dm, ckpt_path=stage1.ckpt)`. Lightning's `ckpt_path` resume expects the same module class — key mismatch between `SupConModule` (backbone+projector) and `SupConFinetuneModule` (backbone+classifier) raises `RuntimeError: Error(s) in loading state_dict` or silently loads the wrong keys.

`SupConFinetuneModule.from_stage1_ckpt()` classmethod filters `backbone.*` keys from a stage-1 checkpoint and calls `freeze_backbone()`. It is never called from `train.py`.

**Fix:** `train.py` must detect `method: supcon_finetune` and route through `SupConFinetuneModule.from_stage1_ckpt(args.ckpt_path, cfg)` instead of the standard dispatcher path.

---

### Warnings (3)

#### WARNING-1 — SupCon stage-1 degraded to SimCLR semantics
**Affects:** SUP-01

`train.py` builds `SSLDataModule` without `sampler_type="class_balanced"`. `SupConModule.training_step` computes `SupConLoss(labels=labels)` — numerically valid but most batches will have singleton classes, making the supervised positive mask equivalent to the self-supervised (SimCLR) positive pair. The `configs/supcon_stage1_resnet18.yaml` comments document this as REQUIRED.

**Fix:** `train.py` must detect `method: supcon` and pass `sampler_type="class_balanced"` to `SSLDataModule`.

---

#### WARNING-2 — Missing exports from `core/__init__.py`
**Affects:** INFRA-04, INFRA-05, FOUND-07

`core/__init__.py __all__` is missing: `PredictorHead`, `SupConLoss`, `MultiCropDataset`, `ssl_collate_multi_crop`, `method_dispatcher`, `register_method`, `available_methods`. All consumers import from submodules directly so no runtime breakage. `from core import PredictorHead` would raise `ImportError`.

---

#### WARNING-3 — Wrong batch-shape comment in tsne_vis.py and umap_vis.py
**Affects:** EVAL-03, EVAL-04

Both scripts contain dead defensive code with the wrong batch-slice index (`imgs[:, 0]` vs correct `imgs[0]`). Harmless because both scripts build their own single-view `ImageFolder` loader and never produce 5-D batches. But a future refactor connecting these scripts to `SSLDataModule` would produce silently incorrect features.
