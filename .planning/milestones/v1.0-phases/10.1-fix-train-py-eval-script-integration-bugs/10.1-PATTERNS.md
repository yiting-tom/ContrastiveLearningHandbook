### `train.py` (CLI entry, request-response) — B3 fix

**Analog:** `train.py` itself (self-edit). The pattern of "construct callbacks list and pass to Trainer" is already established at lines 59-68; we extend it.

**Current callback construction pattern** (`train.py:59-68`):
```python
callbacks: list[L.Callback] = []
if cfg.eval is not None and cfg.eval.knn is not None:
    from eval.knn_callback import KNNCallback
    callbacks.append(KNNCallback(cfg.eval.knn))

trainer = L.Trainer(
    max_epochs=cfg.max_epochs,
    gradient_clip_val=cfg.gradient_clip_val,
    callbacks=callbacks,
)
```

**Pattern to apply (B3 fix per RESEARCH Example C, lines 499-511):**
1. Add to imports (after `import lightning as L`):
   ```python
   from lightning.pytorch.callbacks import ModelCheckpoint
   ```
2. Pre-seed the callbacks list with `ModelCheckpoint(save_last=True, save_top_k=-1)`:
   ```python
   callbacks: list[L.Callback] = [
       ModelCheckpoint(save_last=True, save_top_k=-1),
   ]
   if cfg.eval is not None and cfg.eval.knn is not None:
       from eval.knn_callback import KNNCallback
       callbacks.append(KNNCallback(cfg.eval.knn))
   ```

**Pitfall to enforce** (RESEARCH "Pitfall 3"): Use `save_top_k=-1` (save every epoch) OR pair `save_top_k=1` with `monitor="train/loss"`. Never set `save_top_k=1` without `monitor` — Lightning will warn and produce ambiguous behavior. The recommended config is `save_last=True, save_top_k=-1` (matches RESEARCH Example C and assumption A2).

def test_train_then_linear_probe_e2e(tmp_path, tmp_imagefolder_with_val):
    src = REPO_ROOT / "configs" / "simclr_v1_resnet18.yaml"
    raw = yaml.safe_load(src.read_text())
    raw.update({
        "max_epochs": 1, "batch_size": 4, "num_workers": 0,
        "data_dir": str(tmp_imagefolder_with_val),

    cfg_path = tmp_path / "smoke.yaml"

    train_log_dir.mkdir()

        [sys.executable, str(REPO_ROOT / "train.py"),
         "--config", str(cfg_path)],
        cwd=str(train_log_dir), capture_output=True, text=True, timeout=180,
    )
    assert r1.returncode == 0, r1.stderr

    ckpts = list(train_log_dir.glob("lightning_logs/version_*/checkpoints/*.ckpt"))
    assert ckpts, "train.py produced no checkpoint"

        [sys.executable, str(REPO_ROOT / "eval" / "linear_probe.py"),

        cwd=str(REPO_ROOT), capture_output=True, text=True, timeout=180,

def test_no_ssldatamodule_cfg_misuse():

    eval_dir = REPO_ROOT / "eval"

### Shared Pattern 1: sys.path bootstrap for self-executable subdirectory scripts

**Source:** RESEARCH Pattern 1 (lines 304-311). No in-repo precedent — this is a new convention introduced by Phase 10.1.

**Apply to:** All affected eval scripts (`linear_probe.py`, `finetune.py`, `tsne_vis.py`, `umap_vis.py`, `cam_vis.py`, optionally `dinov2_demo.py` if it imports from `core`/`methods`).

**Exact lines to insert (immediately after `from __future__ import annotations`, before `import argparse` and any project imports):**
```python
import sys
from pathlib import Path
sys.path.insert(0, str(Path(__file__).resolve().parent.parent))
```

**Strict rule (RESEARCH "Pitfall 2"):** MUST be `insert(0, ...)`. NEVER `append(...)` — appending lets a stale system-installed `core` or `methods` package shadow the repo and silently produce wrong behavior.

### Shared Pattern 2: Canonical SSLDataModule construction

**Source:** `train.py:52-57`. This is the ONLY correct way to instantiate `SSLDataModule` in this repo.

**Apply to:** Any code that builds an `SSLDataModule` outside `train.py` (currently the bug sites in `linear_probe.py:221` and `finetune.py:194`).

**Exact pattern:**
```python
dm = SSLDataModule(
    data_dir=cfg.data_dir,
    n_views=cfg.n_views,
    batch_size=cfg.batch_size,
    num_workers=cfg.num_workers,

```

**Anti-pattern (RESEARCH "Anti-Patterns to Avoid"):** Do NOT introduce a wrapper class like `EvalDataModule(cfg)` to "hide" this. Repo has one DataModule and adding indirection violates the established pattern.

### Shared Pattern 3: Lightning checkpoint loading with explicit cfg

**Source:** `eval/linear_probe.py:209-210`, `eval/finetune.py:185-186`, `eval/tsne_vis.py:220-221`, `eval/cam_vis.py:377-378` — all four use the same idiom.

**Pattern (already in use everywhere; do not modify):**
```python
import methods  # noqa: F401  triggers register_method() for all SSL methods
MethodClass = get_method(cfg.method)
model = MethodClass.load_from_checkpoint(args.ckpt, cfg=cfg)
model.eval()
model.to(args.device)
```

**Why explicit `cfg=cfg`:** `BaseSSLModule.__init__` saves `{"cfg": cfg.model_dump()}` (verified at `core/base.py:62`). Lightning's `load_from_checkpoint` would otherwise try to reconstruct `cfg` as a dict; passing the typed `TrainConfig` ensures Pydantic validation fires and downstream attribute access works.

**Phase 10.1 reference only — no change needed.** This pattern is already correct in all five eval scripts and supplies the constraint that the new e2e test must respect.
