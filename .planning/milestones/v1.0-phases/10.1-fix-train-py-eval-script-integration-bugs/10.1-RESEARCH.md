## Summary

Phase 10 marked the documentation-and-tutorial milestone complete and `train.py` itself works (4/4 `tests/test_train_script.py` cases pass). However, the **end-to-end pipeline that the README and tutorial documents — `python train.py --config ... && python eval/<script>.py ...` — does not actually run** because of four concrete integration bugs that the existing `tests/test_eval_integration.py` cannot catch (it bypasses CLI `main()` entry points and constructs everything in-process).

The bugs are mechanical, all in `train.py` and `eval/*.py`, all reproducible in <30s on toy data, and the fixes are localised. No method-level (`methods/*/`) or core (`core/*/`) changes are required. The phase is bounded; the work is editing 5 eval scripts, 1 README/tutorial pass, and adding one true end-to-end smoke test that drives both binaries via subprocess.

**Primary recommendation:** Fix the four documented bugs (sys.path bootstrap, `SSLDataModule(cfg)` constructor mis-call, `last.ckpt` path mismatch, t-SNE perplexity vs n_samples guard), then add `tests/test_e2e_pipeline.py` that subprocess-invokes `train.py` followed by `eval/linear_probe.py` against a toy ImageFolder and asserts the linear-probe script exits 0. Do not touch method modules, core base classes, or Phase 10 documentation prose beyond correcting the broken commands.

## Confirmed Bugs (reproduced this session)

### B1: Eval scripts cannot find the `core` package when run as `python eval/foo.py` [VERIFIED]

**Reproduce:**
```bash
$ python eval/linear_probe.py --help
ModuleNotFoundError: No module named 'core'
```
**Why:** Python adds the *script's directory* (i.e., `eval/`) to `sys.path[0]`, not the repo root. `from core.config import ...` therefore fails. Pytest works because `pytest` adds the rootdir to `sys.path` automatically via `rootdir` discovery.

**Affected files (all 5 eval scripts plus dinov2_demo):**
- `eval/linear_probe.py`
- `eval/finetune.py`
- `eval/tsne_vis.py`
- `eval/umap_vis.py`
- `eval/cam_vis.py`
- (`eval/dinov2_demo.py` likely also affected — check during planning)

**Fix options (recommend Option A):**
- **Option A (recommended, tutorial-friendly):** Add to the top of each eval script (after `from __future__ import annotations`):
  ```python
  import sys
  from pathlib import Path
  sys.path.insert(0, str(Path(__file__).resolve().parent.parent))
  ```
  Pros: zero install ceremony, every documented command in README/tutorial works as-written, no behavioural change for tests. Cons: the path mutation is mildly unidiomatic.
- **Option B (pure-Python idiomatic):** Run via `python -m eval.linear_probe ...`. Requires `eval/__init__.py` (already present), no path hack. Cons: every documented command in README/tutorial/docs would need to be rewritten — this expands the diff into Phase 10 documentation.
- **Option C (packaging idiomatic):** Add `[project]` table to `pyproject.toml` and instruct users to `pip install -e .`. Cons: pyproject.toml currently has only `[tool.pytest.ini_options]` (no project metadata) — adding install metadata is a non-trivial expansion of scope.

**Recommendation:** Option A. It preserves all Phase 10 documentation verbatim and matches what a tutorial reader expects. [CITED: https://alex.dzyoba.com/blog/python-import/]

### B2: `linear_probe.py` and `finetune.py` mis-construct `SSLDataModule(cfg)` [VERIFIED]

**Reproduce:**
```bash
$ PYTHONPATH=. python eval/linear_probe.py <cfg> --ckpt <ckpt> --device cpu
TypeError: expected str, bytes or os.PathLike object, not TrainConfig
  at core/data.py:328 in setup
    train_dir = os.path.join(self.data_dir, "train")
```

**Why:** `SSLDataModule.__init__` signature is `(data_dir: str, n_views: int = 2, batch_size: int = 256, num_workers: int = 4, ...)`. Both `eval/linear_probe.py:221` and `eval/finetune.py:194` call `SSLDataModule(cfg)` — passing the entire `TrainConfig` instance positionally as `data_dir`. Python doesn't reject this (no type enforcement on `__init__`), but `setup()` then calls `os.path.join(TrainConfig_instance, "train")` and crashes.

**Confirmed silent corruption demo:**
```python
dm = SSLDataModule(cfg)
dm.data_dir  # -> entire TrainConfig repr (a TrainConfig object, not a str)
dm.batch_size  # -> 256 (default — cfg.batch_size is silently ignored)
dm.n_views   # -> 2 (default)
```

**Affected files:**
- `eval/linear_probe.py:221`
- `eval/finetune.py:194`
- (Note: `eval/tsne_vis.py` and `eval/umap_vis.py` correctly avoid this by building `ImageFolder` + `DataLoader` directly. `eval/cam_vis.py` walks the filesystem with `os.walk` for raw images — also avoids this bug.)

**Fix:** Use the same explicit-kwarg pattern as `train.py:52-57`:
```python
dm = SSLDataModule(
    data_dir=cfg.data_dir,
    n_views=cfg.n_views,
    batch_size=cfg.batch_size,
    num_workers=cfg.num_workers,
)
```

### B3: Documented checkpoint path `last.ckpt` does not exist by default [VERIFIED]

**Reproduce:**
```bash
$ python train.py --config <cfg>
$ ls lightning_logs/version_0/checkpoints/
epoch=0-step=1.ckpt    # NOT "last.ckpt"
```

**Why:** `train.py` constructs `L.Trainer(...)` with no `ModelCheckpoint` callback configured. Lightning's default `ModelCheckpoint` saves files as `epoch={N}-step={M}.ckpt`. A `last.ckpt` file is only produced when `ModelCheckpoint(save_last=True)` is explicitly passed. [CITED: https://lightning.ai/docs/pytorch/stable/api/lightning.pytorch.callbacks.ModelCheckpoint.html]

**Documented commands that fail:**
- `README.md:43`: `python eval/linear_probe.py configs/simclr_v1_resnet18.yaml --ckpt checkpoints/last.ckpt`
- `README.md:46`: `python eval/umap_vis.py ... --ckpt checkpoints/last.ckpt`
- `docs/tutorial/02_running_an_experiment.md:147,167,181,200,215`: `--ckpt lightning_logs/version_0/checkpoints/last.ckpt`
- `docs/tutorial.md:551,571,585,604,619`: same `last.ckpt` references
- `docs/tutorial/03_comparing_methods.md:65,69`: same pattern

**Fix options (recommend Option A):**
- **Option A (recommended, smaller blast radius):** Add explicit `ModelCheckpoint(save_last=True, save_top_k=1, monitor="train/loss")` callback to `train.py`. Then the documented `last.ckpt` paths Just Work. Single-file change.
- **Option B:** Update README + every tutorial file to use `lightning_logs/version_*/checkpoints/epoch=*-step=*.ckpt` (or shell-glob) — this requires editing 4+ docs and is brittle for users.

**Recommendation:** Option A. One callback added to `train.py` makes all documented commands consistent. [CITED: https://lightning.ai/docs/pytorch/stable/common/checkpointing_basic.html]

### B4: `tsne_vis.py` crashes on small datasets due to perplexity ≥ n_samples [VERIFIED]

**Reproduce:**
```bash
$ PYTHONPATH=. python eval/tsne_vis.py <cfg> --ckpt <ckpt> --device cpu
ValueError: perplexity (30) must be less than n_samples (30)
  at sklearn/manifold/_t_sne.py:_check_params_vs_input
```

**Why:** `TSNEConfig` defaults `perplexities=[10, 30, 50]`. With a small `data_dir` (or smaller-than-config `n_samples`), the per-perplexity loop in `eval/tsne_vis.py:run_tsne` passes raw values directly to `sklearn.manifold.TSNE`. scikit-learn ≥ 1.2 enforces `perplexity < n_samples` strictly. The function in `tests/test_eval_integration.py` works around this by hard-coding `perplexities=[5]`.

**Fix:** In `run_tsne`, after the (optional) PCA pre-reduction, clamp each perplexity:
```python
n = features.shape[0]
safe_perp = max(2, min(perplexity, n - 1))
if safe_perp != perplexity:
    print(f"warn: requested perplexity {perplexity} >= n_samples {n}; clamping to {safe_perp}")
```
Skip a perplexity entirely if `n < 4` (t-SNE needs a minimum sample count to be meaningful).

**Note:** This is a smoke-test reliability fix more than an integration bug per se. With realistic CIFAR-10 (50K samples) the original defaults work fine; the bug surfaces only when users follow the tutorial with a tiny dataset or when the new e2e smoke test runs on the existing `tmp_imagefolder` fixture (15 samples).

### B5 (defensive, recommend including): `linear_probe.py` crashes when `data_dir` has no `val/` split [VERIFIED by code reading]

**Why:** `linear_probe.py:233-240` unconditionally calls `extract_and_cache(model.backbone, dm.val_dataloader(), ...)`. If `data_dir/val/` does not exist, `SSLDataModule.setup()` sets `self.val_dataset = None` and `val_dataloader()` returns `None`. The downstream `for batch in dataloader:` then raises `TypeError: 'NoneType' object is not iterable`.

**Why this matters for Phase 10.1:** The README's CIFAR-10 prep snippet only creates `train/` (no `val/`). So a user following the documented quickstart literally hits this bug *after* B2 is fixed.

**Fix options (planner picks):**
- **Option A:** Document and enforce — raise a clear `FileNotFoundError("Linear probe requires a val/ split. Either add data_dir/val/ or use --val-split fraction-of-train")` if no val split.
- **Option B (more user-friendly):** Auto-split train into 90/10 train/val when `val/` is missing. Lighter coupling but adds non-trivial code.
- **Option C (smallest fix):** Update README CIFAR-10 prep snippet to also generate a `val/` split (~5K images). Code change is purely documentation.

**Recommendation:** Combine A + C — fail fast with a helpful error AND fix the documented prep snippet. The error path keeps users out of the silent-corruption trap; the doc fix removes the friction for the canonical example.

### Pattern 1: sys.path bootstrap for self-executable scripts in subdirectories

**What:** A 3-line preamble that lets a script in a subdirectory of a non-installed project import sibling top-level packages.

**When to use:** Any script invoked as `python <subdir>/<script>.py` from a project root that does not have an editable install.

**Example:**
```python
# Source: https://alex.dzyoba.com/blog/python-import/
# Add this AFTER `from __future__ import annotations` and BEFORE any
# `from core...` or `from methods...` imports in each eval/*.py.
import sys
from pathlib import Path
sys.path.insert(0, str(Path(__file__).resolve().parent.parent))
```

The `Path(__file__).resolve().parent.parent` chain resolves the eval script -> `eval/` dir -> repo root, regardless of how the user cwd's into the project.

### Pattern 2: Subprocess-driven end-to-end test (the missing test type)

**What:** A pytest test that uses `subprocess.run([sys.executable, "train.py", ...])` followed by `subprocess.run([sys.executable, "eval/linear_probe.py", ...])` and asserts both exit 0 and produce expected files.

**When to use:** This is the canonical way to verify CLI integration that bypasses Python-level import shortcuts that pytest gives you for free.

**Example:**
```python
# Source: pattern derived from existing tests/test_train_script.py:test_train_py_help_exits_zero
# combined with tests/test_eval_integration.py fixture style.
import subprocess, sys, yaml
from pathlib import Path

REPO_ROOT = Path(__file__).resolve().parent.parent

def test_train_then_linear_probe_e2e(tmp_path, tmp_imagefolder_with_val):
    # 1. Write override config that points at the toy ImageFolder
    src = REPO_ROOT / "configs" / "simclr_v1_resnet18.yaml"
    raw = yaml.safe_load(src.read_text())
    raw.update({
        "max_epochs": 1, "batch_size": 4, "num_workers": 0,
        "data_dir": str(tmp_imagefolder_with_val),
    })
    cfg_path = tmp_path / "smoke.yaml"
    cfg_path.write_text(yaml.safe_dump(raw))

    # 2. Train (subprocess so we exercise sys.path / argparse / ModelCheckpoint)
    train_log_dir = tmp_path / "train_logs"
    r1 = subprocess.run(
        [sys.executable, str(REPO_ROOT / "train.py"),
         "--config", str(cfg_path)],
        cwd=str(train_log_dir), capture_output=True, text=True, timeout=180,
    )
    assert r1.returncode == 0, r1.stderr

    # 3. Locate the checkpoint Lightning produced
    ckpts = list(train_log_dir.glob("lightning_logs/version_*/checkpoints/*.ckpt"))
    assert ckpts, "train.py produced no checkpoint"

    # 4. Linear probe (subprocess so we exercise sys.path bootstrap fix)
    r2 = subprocess.run(
        [sys.executable, str(REPO_ROOT / "eval" / "linear_probe.py"),
         str(cfg_path), "--ckpt", str(ckpts[0]), "--device", "cpu"],
        cwd=str(REPO_ROOT), capture_output=True, text=True, timeout=180,
    )
    assert r2.returncode == 0, r2.stderr
    assert "Final val/acc:" in r2.stdout
```

This test would have caught all four bugs. Mark it `@pytest.mark.slow`.

### Anti-Patterns to Avoid

- **DO NOT** monkey-patch `Trainer.__init__` from the e2e test. The whole point is to drive the real binary; a mock defeats the purpose. Use real subprocess + small `max_epochs=1` + `limit_train_batches=1` (passed via the YAML override, NOT via monkeypatch).
- **DO NOT** add `python -m pip install -e .` to the docs as the fix for B1. It's a 30-second fix to add 3 lines to each script and zero install ceremony for tutorial readers.
- **DO NOT** weaken `extra='forbid'` on Pydantic configs to "fix" cfg-passing issues. The constructor calls are the bug, not the schema.
- **DO NOT** add a wrapper class like `EvalDataModule` that takes a `cfg`. The codebase has one DataModule (`SSLDataModule`); adding a second tier of indirection violates the established pattern. Just call `SSLDataModule` with the same kwargs `train.py` uses.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Find a sibling package from a script | Custom `importlib` machinery | `sys.path.insert(0, str(Path(__file__).resolve().parent.parent))` | Universally understood Python idiom, 1 line, no edge cases |
| Discover the latest checkpoint | `glob` + `max(by mtime)` walks in every eval script | `ModelCheckpoint(save_last=True)` in `train.py` produces a stable `last.ckpt` | Lightning provides this for free; rolling your own re-implements `ModelCheckpoint` |
| Validate t-SNE perplexity vs sample count | Re-implement scikit-learn's check before calling | Just clamp `min(perplexity, n_samples - 1)` and let scikit-learn run | One-liner; the official advice from sklearn docs |
| End-to-end CLI smoke test | Mock `sys.argv` + monkeypatch `Trainer` | `subprocess.run([sys.executable, "script.py", ...])` | Exercises argparse, sys.path, file IO, and exit codes — exactly what we need to catch B1-B4 |

**Key insight:** Every bug in this phase exists because the existing tests bypass exactly the layer where the bug lives. `tests/test_train_script.py` monkeypatches `Trainer` (so it never runs Lightning's actual checkpoint logic → B3 hidden). `tests/test_eval_integration.py` constructs everything in-process (so it never invokes `python eval/foo.py` → B1, B2 hidden) and constructs feature dataloaders manually (so it never calls `SSLDataModule(cfg)` → B2 also hidden). The fix is not just patching the bugs, but adding the missing test layer (subprocess-driven end-to-end) so they cannot regress.

## Common Pitfalls

### Pitfall 1: Fix B2 in only one of the two affected scripts
**What goes wrong:** `linear_probe.py` is fixed but `finetune.py` is missed (or vice versa). Smoke test passes for the linear probe path; finetune still crashes for the next user.
**Why it happens:** The two files have nearly-identical bug sites at different line numbers; a search/replace might miss one.
**How to avoid:** Add a regression assertion: `grep -n 'SSLDataModule(cfg)' eval/` should return nothing after the fix. Add a smoke test for `finetune.py` even though success criterion 3 only mandates `linear_probe.py`.
**Warning signs:** A planner task that mentions only `linear_probe.py` in its description. The plan needs to enumerate both files.

### Pitfall 2: Fix B1 with `sys.path.append` instead of `sys.path.insert(0, ...)`
**What goes wrong:** If the user has a stale `core` or `methods` package installed elsewhere on sys.path, `append` lets the stale install win. Subtle silent wrong behaviour.
**Why it happens:** `append` and `insert(0, ...)` look interchangeable but are not.
**How to avoid:** Always `insert(0, ...)` for repo-bootstrap shims. Explicitly say so in the planner instructions.
**Warning signs:** Code review finds `sys.path.append` — flag immediately.

### Pitfall 3: Add `ModelCheckpoint(save_last=True)` without `save_top_k`
**What goes wrong:** Default `save_top_k=1` requires a `monitor` metric. If unset, Lightning logs a warning and saves only `last.ckpt` (which is fine for tutorial use), but the warning is noisy. If `monitor` is set to a metric that doesn't exist (e.g., `val/loss` when there's no val loop in pretraining), it crashes.
**Why it happens:** ModelCheckpoint has subtle interactions between `save_top_k`, `save_last`, and `monitor`.
**How to avoid:** Use `ModelCheckpoint(save_last=True, save_top_k=-1)` (save every epoch) OR `ModelCheckpoint(save_last=True, save_top_k=1, monitor="train/loss")` (save best by train loss + last). Verify the chosen logging key exists in the SimCLR module.
**Warning signs:** Warnings about "monitor metric not found" in train logs after the fix.
**Reference:** [CITED: https://lightning.ai/docs/pytorch/stable/api/lightning.pytorch.callbacks.ModelCheckpoint.html]

### Pitfall 4: e2e smoke test takes >2 minutes and slows CI
**What goes wrong:** The new `tests/test_e2e_pipeline.py` is too slow because Lightning startup + ResNet18 init + 1 batch is ~10–15 seconds, and subprocess overhead doubles it.
**Why it happens:** Test isn't optimised for the CI loop.
**How to avoid:** (a) mark the test `@pytest.mark.slow` so it runs only on phase-gate not per-task; (b) override the YAML to `max_epochs: 1`, `limit_train_batches: 1` (via Lightning), `batch_size: 2`, `num_workers: 0`, `n_views: 2`; (c) use the existing `tmp_imagefolder` fixture (3 classes × 5 images) but extended with a tiny `val/` split.
**Warning signs:** Test takes longer than ~30 seconds in CI on a single core.

### Pitfall 5: README/tutorial doc fix breaks the original phase 10 verification
**What goes wrong:** Phase 10 verification scored 5/5 partly on "all referenced configs/eval scripts/CLI commands resolve". Editing the docs to fix B3 might leave dangling phrases.
**Why it happens:** Docs and code change in lockstep but Phase 10 verification was done as a snapshot.
**How to avoid:** Either (a) add `save_last=True` so docs stay correct (Option A path — recommended), OR (b) re-grep all the `last.ckpt` references after editing and confirm they all point to a real, produced filename. Do not re-run the full Phase 10 verification — Phase 10.1 has its own success criteria.
**Warning signs:** A new doc references a path/file that does not exist after the fix.
