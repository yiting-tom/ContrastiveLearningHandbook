# Phase 10.1: Code Review Report

**Reviewed:** 2026-05-03
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Phase 10.1 addresses five integration bugs (B1–B5) between `train.py` and the
`eval/` scripts. The `sys.path` bootstrap (B1), `SSLDataModule` keyword-argument
fix (B2), `ModelCheckpoint(save_last=True)` wiring (B3), t-SNE perplexity
clamping (B4), and the missing-`val/`-split guard (B5) are all implemented
correctly and are backed by regression tests.

However, two bugs were introduced or left unresolved that will cause incorrect
behavior or crashes at runtime:

1. `eval/linear_probe.py` uses the wrong tensor slice (`imgs[:, 0]`) when
   handling the 5D multi-view batches produced by `ssl_collate_fn`. This
   produces corrupted feature extraction and a shape-mismatch crash when
   building the `TensorDataset`.

2. `eval/tsne_vis.py` and `eval/umap_vis.py` call `ImageFolder(root=cfg.data_dir)`
   directly without replicating `SSLDataModule`'s train/-subdir auto-detection.
   When `data_dir` points to a `train/`+`val/` parent directory, ImageFolder
   silently treats the split names as class names, producing semantically wrong
   visualizations. The slow-path tests work around this with a separate config
   but the bug is not fixed in the scripts themselves.

## Critical Issues

### CR-01: Wrong tensor slice for multi-view batches in `extract_and_cache` — crash + data corruption

**File:** `eval/linear_probe.py:83-84`

**Issue:**
`ssl_collate_fn` (in `core/data.py`) stacks views as
`[n_views, B, C, H, W]` — the view dimension is **dim 0**, the batch
dimension is **dim 1**. `extract_and_cache` receives this tensor as `imgs`,
checks `imgs.ndim == 5` correctly, but then slices `imgs[:, 0]` which indexes
along dim 1 (batch) and returns `[n_views, C, H, W]` — only the views of the
**first sample** in the batch.

The backbone then processes these `n_views` items thinking they are a batch of
`n_views` images, appending `[n_views, feat_dim]` to `all_features`. Labels
are appended as `[B]` per batch. After the loop:

```
torch.cat(all_features).shape  # [n_batches * n_views, feat_dim]
torch.cat(all_labels).shape    # [n_batches * B, ]
```

With `n_views=2` and `B=4`, features are 2× shorter than labels.
`TensorDataset(train_feats, train_labels)` raises `RuntimeError: tensors must
have the same number of rows`. The script crashes before training.

The accompanying inline comment compounds the confusion by describing the
shape as `[B, n_views, C, H, W]` — the opposite of what `ssl_collate_fn`
actually produces.

**Fix:**
```python
# core/data.py ssl_collate_fn output shape: [n_views, B, C, H, W]
# Dim 0 is the view axis; imgs[0] picks view 0 for all B samples → [B, C, H, W].
if isinstance(imgs, torch.Tensor) and imgs.ndim == 5:
    imgs = imgs[0]  # was: imgs[:, 0]
```

Update the comment on the preceding line:
```python
# Handle SSL multi-view batches: ssl_collate_fn produces shape [n_views, B, C, H, W].
```

### CR-02: `tsne_vis.py` and `umap_vis.py` do not auto-detect `train/` subdir — silent wrong class labels

**File:** `eval/tsne_vis.py:262`, `eval/umap_vis.py:230`

**Issue:**
Both scripts build their dataset with:
```python
dataset = ImageFolder(root=cfg.data_dir, transform=transform)
```

`SSLDataModule.setup()` correctly probes for a `train/` subdirectory and uses
it if present:
```python
train_dir = os.path.join(self.data_dir, "train")
if not os.path.isdir(train_dir):
    train_dir = self.data_dir
```

The visualization scripts perform no such check. When the user follows the
Quickstart (which produces `data/cifar10_imagefolder/` with `train/` and `val/`
subdirs and stores that path in the YAML), `ImageFolder` will discover two
"classes" named `train` and `val` instead of the actual 10 CIFAR classes.
The scatter plots will show a meaningless binary separation. No error is raised;
the output PNG looks plausible but is semantically wrong.

The slow-path regression tests in `test_e2e_pipeline.py` mask this by writing
a second `vis_smoke.yaml` whose `data_dir` points directly at the `train/`
subdir. That workaround is invisible to end users following the README.

**Fix — replicate `SSLDataModule`'s auto-detection in both scripts:**

In `eval/tsne_vis.py` `main()`, replace line 262:
```python
# Auto-detect train/ subdirectory (mirrors SSLDataModule.setup logic)
data_root = Path(cfg.data_dir)
train_sub = data_root / "train"
img_root = train_sub if train_sub.is_dir() else data_root
dataset = ImageFolder(root=str(img_root), transform=transform)
```

Apply the identical change in `eval/umap_vis.py` `main()` at line 230.

## Warnings

### WR-01: Duplicate `from pathlib import Path` import in `cam_vis.py`

**File:** `eval/cam_vis.py:21` and `eval/cam_vis.py:26`

**Issue:**
`from pathlib import Path` appears twice. The first occurrence (line 21) is
inside the `sys.path.insert` bootstrap block; the second (line 26) is the
normal module-level import. The duplication is harmless at runtime but
misleading: it implies the first import is only needed for the bootstrap and
might be pruned incorrectly during a future refactor.

**Fix:** Remove line 21 (`from pathlib import Path`) so the only import is the
one at line 26, which is already in scope for the `sys.path.insert` call on
line 22 because Python imports are resolved at module load time in statement
order — but more cleanly, restructure so line 22 references the single import:

```python
# At the top of the file, before any other imports:
import sys
from pathlib import Path

sys.path.insert(0, str(Path(__file__).resolve().parent.parent))

import argparse
import warnings
# ... rest of imports
from pathlib import Path   # <-- REMOVE this second import
```

### WR-02: `open()` without context manager in `tsne_vis.py` and `umap_vis.py`

**File:** `eval/tsne_vis.py:234`, `eval/umap_vis.py:201`

**Issue:**
Both scripts open the config file inline without a context manager:
```python
cfg = TrainConfig.model_validate(yaml.safe_load(open(args.config)))
```
The file handle is not explicitly closed. CPython's reference-counting GC
will close it immediately (when the expression is evaluated), but PyPy and
other implementations may keep it open for an indeterminate period. This also
differs from the pattern used by the other three eval scripts (which all use
`with open(...) as f:`), making the codebase inconsistent.

**Fix:**
```python
with open(args.config) as f:
    cfg = TrainConfig.model_validate(yaml.safe_load(f))
```

### WR-03: Incorrect shape comment in `tsne_vis.py` and `umap_vis.py`

**File:** `eval/tsne_vis.py:67`, `eval/umap_vis.py:62`

**Issue:**
Both `extract_features` functions contain the comment:
```python
# Handle multi-view augmented batches (shape [B, n_views, C, H, W])
```
`ssl_collate_fn` produces `[n_views, B, C, H, W]`, not `[B, n_views, C, H, W]`.
The comment is wrong. This matters because the code that follows (`imgs[:, 0]`)
is dead code in these functions (the DataLoaders built inline use standard
`ImageFolder`, producing 4D batches, so `imgs.ndim == 5` is never `True`).

If a caller ever passes an `SSLDataModule`-backed DataLoader to
`extract_features`, the dead branch would execute and slice incorrectly for
the same reason as CR-01.

**Fix:** Either remove the dead branch entirely (since these scripts always use
4D batches from `ImageFolder`), or correct both the comment and the slice to
match the real shape:
```python
# Handle multi-view batches: ssl_collate_fn produces [n_views, B, C, H, W]
if imgs.ndim == 5:
    imgs = imgs[0]  # take first view → [B, C, H, W]
```

### WR-04: `test_no_ssldatamodule_cfg_misuse` pattern is too narrow

**File:** `tests/test_e2e_pipeline.py:131`

**Issue:**
The regression test greps for the exact strings
`"dm = SSLDataModule(cfg)"` and `"SSLDataModule(cfg)"`. This misses
alternative variable names (`data_module = SSLDataModule(cfg)`) and
indented contexts. Since `stripped.startswith(...)` is used, an expression
preceded by any assignment prefix other than `dm = ` bypasses the gate. The
test provides a false sense of completeness.

**Fix:** Match the pattern on the argument rather than the full expression:
```python
import re
# Catches any positional-arg call: SSLDataModule(cfg) or SSLDataModule(cfg, ...)
# Skips keyword-arg calls: SSLDataModule(data_dir=cfg.data_dir, ...)
if re.search(r'\bSSLDataModule\s*\(\s*cfg\b', stripped):
    bad.append(f"{py.name}:{i}: {stripped}")
```

**File:** `eval/linear_probe.py:82`

```python
# Handle SSL multi-view batches: batch[0] has shape [B, n_views, C, H, W]
```

`ssl_collate_fn` produces `[n_views, B, C, H, W]`. The axes are swapped.

```python
# Handle SSL multi-view batches: ssl_collate_fn produces [n_views, B, C, H, W].

```

### IN-02: `val_loader` variable shadowing in `linear_probe.py`

**File:** `eval/linear_probe.py:252` and `eval/linear_probe.py:280`

**Issue:**
The name `val_loader` is used twice for two completely different objects: first
for the `SSLDataModule` raw-image DataLoader used to extract features, then
immediately overwritten with a `TensorDataset`-backed DataLoader for linear
head training. The logic is correct (the first loader is consumed before the
second assignment), but the shadowing obscures intent and makes the code harder
to reason about.

**Fix:** Rename the first variable:
```python
val_raw_loader = dm.val_dataloader()
if val_raw_loader is None:
    raise FileNotFoundError(...)

val_feats, val_labels = extract_and_cache(
    model.backbone,
    val_raw_loader,   # ← renamed
    ...
)
```
